# ZEN Study 学習統計 拡張機能 — ダッシュボード設計

> 前提: `API_REFERENCE.md` の取得可能データを元に、自分の学習データを詳細な統計UIで表示する個人用 Chrome 拡張。read-only・私的使用の範囲。

---

## 【第一原則】絶対に守る制約（最優先・例外なし）
ZEN Study は N高／S高等の**正式な授業**で、文部科学省の単位認定等が関わる。成績・学習進捗は公的記録に直結するため、以下を最優先で厳守する。

1. **HTTP は GET のみ。** POST / PUT / PATCH / DELETE 等、**サーバの状態を変更する通信は一切行わない**（完全 read-only）。
2. **動画の不正視聴・進捗の自動進行を行わない。** 特に**進行中の課題動画の視聴進捗を自動で進める**行為は絶対禁止。視聴フラグ・進捗%・レポート状態など、**あらゆる学習記録を書き換えない／偽装しない**。
3. **本ツールは「表示専用」。** 既に取得できるデータを見やすく可視化するだけ。学習の実績を1ミリも変えない。
4. コンテンツ(動画/教材)本体のダウンロード・保存もしない（ToS 第9条）。

> この原則に反する機能は、たとえ技術的に可能でも実装しない。設計・実装の各段階でこの節に照らして判断する。

## 1. 全体アーキテクチャ（Manifest V3）

```
┌─ Service Worker (background) ─────────────────────┐
│  chrome.alarms で 1日1回                           │
│   → GET /v2/learning_amounts をスナップショット      │
│   → IndexedDB に日別データを追記（14日制限を突破）      │
└───────────────────────────────────────────────────┘
┌─ Content Script  (www.nnn.ed.nico/setting) ───────┐
│  ★描画方式 = DOM注入（採用）                        │
│  Shadow DOM ルートをページに append し、その中に      │
│  統合UIを描画（既存「学習数」カードを高精細版に置換 +  │
│  「詳細」トグルでインライン展開）。                    │
│  各APIを Cookie付き fetch → Shadow DOM内でチャート描画 │
│  IndexedDB の長期履歴 + ライブ取得データを合成         │
└───────────────────────────────────────────────────┘
```

### 描画方式: DOM注入（決定）
`/setting` ページ内にその場で表示する。独立タブは作らない。
- **Shadow DOM 必須**: 自前ルートを `append` し全UIをその中に閉じ込める（React管理下に差し込まない＝再描画で消えない、CSS相互衝突なし）。
- **SPA対応**: `MutationObserver` + `history.pushState/replaceState` パッチで遷移を検知し、`/setting` に来たら再注入（idempotent に）。
- **アンカーはテキスト/aria/構造**でDOMの差し込み位置を特定。**ハッシュ化クラス名は狙わない**（デプロイで壊れる）。差し込み位置が見つからなければ安全にフォールバック（ページ末尾に自前セクション）。
- **数値はAPI(GET)から取得**、DOMは位置の目印だけ（DOMスクレイピングしない）。
- **場所の制約**: 既存カードは小さい → コンパクト置換 + 「詳細」インライン展開で全カードを収める。
- トレードオフ: フロント更新で注入点が壊れうる（保守コスト高め）。ただし表示専用ゆえ危険はなく、フォールバックで最低限の表示は維持。

### manifest.json 要点
- `manifest_version: 3`
- `host_permissions`: `https://api.nnn.ed.nico/*`, `https://www.nnn.ed.nico/*`
- `permissions`: `storage`, `alarms`
- `background.service_worker`
- `content_scripts`: matches `https://www.nnn.ed.nico/*`（SPAなのでURL単位でなく全体に注入し、内部でルート判定）。`run_at: document_idle`
- `action`（任意: オン/オフや設定のトグル用ポップアップ。メイン表示はDOM注入なのでダッシュボードタブは不要）

### 本家デザイントークン（実測・見た目を合わせる）
`/setting` の computed style から採取。`src/styles.ts` に反映済み。
- **font**: `-apple-system, BlinkMacSystemFont, "Hiragino Kaku Gothic Pro", ZaneYuGothicM, Meiryo, sans-serif`（`ZaneYuGothicM`はZEN独自。@font-faceは文書スコープなのでShadow内でも効く）
- **ブランド青**: `#0077d3`（strong `#005596`）／**ink** `#222222`／**muted** `#828282`
- **カード**: 白 / `border-radius: 8px` / border無し / `box-shadow: rgba(0,0,0,.1) 0 1px 1px, rgba(0,0,0,.2) 0 0 2px`
- ダークは本家に無いので自前（primary `#4aa3ee`）。フラットバーは contrast≥3:1 を light/dark で確認済（逐次ランプはM2で正式validate）。

### データ取得
- fetch は `credentials:'include'` でログインCookieを送る。host_permissions があれば api サブドメインへ cross-origin 取得可。
- **レート配慮(ToS 第12条9)**: フル取得はダッシュボードを開いた時のみ。日次スナップショットは1日1回。結果は IndexedDB にキャッシュし、`revision`/更新検知で差分のみ再取得。

### 保存
- **IndexedDB**: 日別学習数の時系列（自前蓄積）、コース/章/教材の進捗スナップショット履歴。
- **chrome.storage.local**: 設定（テーマ・表示するカード・最終取得時刻）。

### チャートライブラリ
- CSP のため**ローカル同梱**（CDN不可）。候補: uPlot(軽量・時系列向き) / Chart.js(手軽) / ECharts(高機能・レーダーやヒートマップ内蔵)。
- → **ECharts 推奨**（レーダー・カレンダーヒートマップ・ツリーマップを標準で持つ）。軽さ優先なら uPlot + 自前ヒートマップ。

## 2.5 「学習数」カードの統合設計（14日窓ジレンマの結論）

2案のモック（`mockups/mockup-a-immediate.html` / `mockup-b-growth.html`）を実レンダリング比較した結論。

**核心の兼ね合い**: 14日窓は「線には短く点には多すぎる中間」。
- 案A(即時充実)=初日から密で完成して見えるが、窓固定で**育たない**（累計は単調・曜日は常に2サンプル）。
- 案B(長期蓄積)=3ヶ月後は圧巻だが、**初日は本質的に薄い**（カレンダー疎・MA<7日は無意味）。

→ **二者択一にせず、時間軸で主役を交代させる「1枚に統合」**を採用。

### 技術的な後押し（Bの弱点の緩和）
学習数APIは毎回「直近14日」を返す。よって **拡張が14日以内に一度でも起動すれば、取りこぼした日を遡って補完可能**（`daily_amount`で照合、`total_amount`の差分でも検算）。→ Bの「スナップショット漏れ＝永久の穴」は **14日を超えて起動しなかった場合のみ**発生。実運用ではほぼ塞げる。

### 統合カードの構成（上から）
1. **ヘッダ + 「記録中·N日目」バッジ**（Bの正直な枠組み。空白＝経過であって故障でないと明示）
2. **KPI行**: 累計 / 2週平均 / **ストリーク** / 最高日（A+BのKPI統合。ストリークは14日でも成立）
3. **直近14日バー + 2週平均線**（A。初日から確実にリッチな中心。ホバーで平均比±）
4. **学習カレンダー（ヒートマップ）**（B。14日を種に、日が経つほど濃くなる長期の主役。セル4状態=実データ/0/未記録/未来）
5. **長期トレンド + 7日MA**（B。日/週/月トグル。7日以降で安定）
6. **曜日別リズム**（A。初日は2サンプルだが、履歴が増えるほど統計的厚みが増す“橋渡し”要素）

### refinement
- Bの**「3ヶ月後(予測)」トグルは常設しない**。予測ダミーを統計ツールに常時混ぜるのは誤読の元。→ **初回起動時のオンボーディング・プレビュー限定**（「続けるとこう育ちます」）に格下げ。
- 時間経過で自然と 4→5 が主役になり、3 は「直近の詳細」ビューとして残る。役割交代が起きる。
- カラーは両案とも sequential 単一ヒュー(青)で validate 済み。統合後もランプを共有し light/dark 再検証する。

## 3. 14日制限への対処（重要）
- API は日別を直近14日しか返さない。
- **初回起動時**: 取得できる14日分をバックフィルして時系列の初期値にする。
- **以後**: 日次アラームで毎日1点を追記。累計値(total_amount)の差分でも日別を補完可能（取りこぼし対策）。
- これにより G/H/I/J は日が経つほど濃くなる。

## 4. 段階的な実装ロードマップ
1. **MVP**: manifest + content script（Shadow DOM注入の土台 = 注入/再注入/フォールバック）+ 統合「学習数」カード（初日データ: 累計/KPI/14日バー+平均線/曜日別）。
2. **蓄積基盤**: service worker + IndexedDB + 日次スナップショット → カレンダー/トレンド/ストリークを統合カードのインライン展開に追加。
3. **他カード**: A(コース進捗)/D(バーンダウン)/E(スコアレーダー) を「詳細」展開に追加。
4. **深掘り**: B/C/**M**（章・教材の一括取得＋キャッシュ、レート配慮）。
   - ⚠️ **M/B/C は全章の⑥を舐めるため通信数が多い**（9コース×各6〜12章＝数十リクエスト）。第一原則＆第12条9に従い: 初回のみ全取得、以後は IndexedDB キャッシュ＋`progress`変化検知で差分のみ、逐次(直列/低並列)取得・進捗表示付き。バックグラウンドで一気に叩かない。
5. **仕上げ**: 設定（カード表示切替・テーマ）、オンボーディング（3ヶ月後プレビュー）、K/L。

## 5.5 サイト全体ダークモード（我々の手作り実装・確定）
本家にダークが無いので自前実装。方式を実サイトで比較検証した結果:
- ~~filter反転（invert+hue-rotate）~~ → 手軽だが**配色が不自然**（ユーザー却下）。
- **動的リマップ（採用）**: 要素の computed style を見て、白いサーフェス→暗色(2段)、暗い文字→明色(2段)、`background-image`→brightnessダウン。**ブランド青(#0077d3系)は残す**。ハッシュ化クラスに非依存。
- トークン: BG `#14181d` / S1 `#1b2027` / S2 `#232b35` / INK `#e6ebf1` / MUTED `#9aa6b2` / BORDER `#2c333d`。
- MutationObserver で新規ノードを追従（有効時のみ、rAFでバッチ）、`chrome.storage.local` で永続化。
- **トグルは本家ナビ項目を複製注入**（`<a><i type><span>` を cloneNode でクラスごと継承→ネイティブな見た目）。アイコンは月/太陽の線SVG(currentColor)で、本家`<i>`の実寸(28px)に合わせラベル縦位置を揃える。ラベル「テーマ」。ナビ無しページはフローティングにフォールバック。React再描画対策で定期(1.5s)＋遷移時に再設置。
- 自前カードは反転に乗せず、`data-theme="dark"` で**自前のダーク配色**（面 `#1b2027` = サイトのリマップ面と一致）。
- 既定はライト固定（本家に合わせ prefers-color-scheme に追従しない）。

### 堅牢化（実サイト検証で判明した不具合の対策）
- **ヘッダ白化**（React再マウントで tag class が消え白に戻る）→ **構造セレクタ `header,[role="banner"]` に直接CSS**。class非依存で常に暗色。
- **ロゴ視認性**（ZEN Study/Apple等の暗色ロゴが暗面で消える）→ `img[src*=".svg"]` を invert+hue-rotate で明色化。**アバターはjpeg/pngなので対象外**。
- **バナー文字が暗すぎ**（`filter:brightness` が文字も暗くする）→ 大きな背景画像(≥300×50)は `.zss-ov` の**暗幕オーバーレイ(::before)**にし、`> *{z-index}` で文字は明るいまま。
- **/home等でサーフェスが白いまま/取りこぼし**（1回スキャン後に描画される要素）→ MutationObserver を **childList + attributes(['class','style'])** に拡張（class変更で貼り直し）＋**ルート遷移時の再スキャン**(`rescanSoon`, 250/900ms)。
- **/settingへ遷移でカードが消える**（アンカー未描画時に body へ仮挿入→再描画で消失）→ 注入は**本家カード発見時のみ**(`applyOverwrite`、body仮挿入を廃止)。毎tick冪等で、本家復活時は再度隠し自前カードが外れたら貼り直す。
- **遷移時の白フラッシュ**（新ページが暗色化される前に一瞬白）→ ルート遷移時に**暗幕ベール**(`#zss-veil`)を一瞬被せ、スキャン完了頃(≈140ms)にフェードアウト。
- **未解放動画行の明るいハッチ**（`::after` の `fill_disabled_stripe.svg` がダークで浮く）→ 行サイズ要素に限り `::after` を検査し `.zss-hatch` を付与、`filter: invert()` で**暗いハッチに反転**。視聴済み/視聴可能/未解放の3状態が明確に。

## 5. 設計上の判断（すべて確定）
- **描画方式**: **DOM注入**（`/setting` 内に Shadow DOM で表示。§1参照）。
- **フレームワーク/ビルド**: **素の TypeScript + Vite**（軽量・Shadow DOMと好相性）。
- **チャート**: **uPlot（時系列）+ 自前SVG（バー/ヒートマップ/レーダー/曜日別）**。ローカル同梱、CDN不可。
- **配布**: **個人利用のみ**（私的使用の範囲でToSリスク最小）。ただしコードは将来公開できる素性に保つ。
